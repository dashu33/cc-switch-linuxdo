# 仓库协作约束

## 自用特性保存方案（强制）

本仓库自用改动采用「四层真相源」：

1. **行为真相**：`自用特性/<中文特性名>/README.md`
2. **定位真相**：`自用特性/<中文特性名>/CODEMAP.md`
3. **代码真相**：项目原有生产路径中的唯一实现
4. **变更真相**：按特性拆分的 Git 提交；可选 `patches/` 仅作灾备导出

### 文档层

- 任何新增特性，或会改变已有特性的用户行为、交互、配置、数据结构、接口、兼容性或验证方式的修改，必须在同一任务中创建或更新 `自用特性/<中文特性名>/`。仅行为不变的重构、格式化、注释和依赖维护可豁免。
- `<中文特性名>` 必须简短、明确、稳定。同一特性始终复用原目录，不得按版本或小改动重复创建目录；不得新增英文特性目录。
- 每个特性目录至少包含：
  - `README.md`：目标与范围、用户行为、关键决策、非目标、兼容或迁移说明、验收标准与回归清单。
  - `CODEMAP.md`：相关实现、测试、配置、资源的仓库相对路径，关键符号、数据流、易冲突点和验证命令。
- 截图、示例、迁移说明、专用脚本等特性材料也应放入对应中文特性目录。
- 生产源码仍放在项目原有目录；在 `CODEMAP.md` 中引用真实路径。禁止为了满足文档要求复制生产源码，避免双份实现漂移。
- 新建、合并或重命名特性目录时，必须同步更新 `自用特性/README.md` 索引；实现、测试、README、CODEMAP 必须在同一任务内保持一致。
- 功能实现文件或关键符号发生变化时，即使外部行为不变，也必须同步更新已有特性的 `CODEMAP.md`。
- 缺少对应特性文档、代码地图或索引更新时，新增/更新特性不得标记为完成。
- `自用特性/` 是自用功能文档的唯一维护位置；不得重新创建旧 `FEATURE/` 文档树。

### 代码与提交层

- 生产实现只保留一份，路径以 `CODEMAP.md` 为准；不得在 `自用特性/` 下复制可运行源码。
- 自用特性相关提交应尽量按特性拆分，推荐前缀：
  - `feat(自用/<中文特性名>): ...`
  - `fix(自用/<中文特性名>): ...`
  - `docs(自用/<中文特性名>): ...`
- 禁止把多个无关自用特性揉进同一个杂项提交；格式化、依赖升级、CI 调整应与功能提交分离。
- 大特性开发可使用 `self/<中文特性短名>` 或 `codex/<topic>` 分支，合并回自用主线前应整理为可读的特性提交序列。

### 变更备份层（可选）

- `patches/` 不是日常开发主路径，只作为灾备或迁移材料。
- 需要导出时，按上游基线生成 patch 系列，例如：
  - `git format-patch <upstream-baseline>..HEAD --output-directory patches`
- patch 文件若入库，必须与当前自用提交语义一致；过期 patch 应及时重导或删除，避免被误当成可运行源码。

### 上游对齐

1. 保存当前工作树，确认无未提交关键改动。
2. 阅读 `自用特性/README.md`，确认仍需保留的特性。
3. 按各特性 `CODEMAP.md` 核对真实文件与关键符号是否被上游移动。
4. 优先 `rebase` 自用提交到新上游；冲突按特性逐个解决。
5. 运行各特性回归清单与验证命令。
6. 若行为、路径或符号变化，同一任务内更新 README、CODEMAP 与总索引。
7. 如使用 `patches/`，升级完成后按需重导。

## 完成前检查

1. 确认相关中文特性目录存在且名称稳定。
2. 确认 README 的行为和验收标准与实现一致。
3. 确认 CODEMAP 列出所有生产文件、测试、i18n、迁移、易冲突点和验证命令。
4. 确认 `自用特性/README.md` 已更新。
5. 确认相关提交按特性拆分，未把无关改动混进同一提交（若本任务产生提交）。
6. 运行与风险相称的测试，并把特殊环境限制记录到特性 README。
7. 若导出了 `patches/`，确认其只作备份且不与生产实现双写。
