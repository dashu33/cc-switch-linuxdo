# 自用特性保存方案

> 文档状态：最后同步 2026-07-16  
> 目录：`自用特性/自用特性保存方案/`（由原「自用特性文档规范」扩展并更名）

## 目标

把本分支相对原作者代码的自用差异，保存成可升级、可回退、可核对的体系，而不是第二份源码仓。

核心目标：

- 上游更新时，能快速判断哪些自用行为仍要保留
- 冲突时，能快速定位真实代码落点
- 回退或迁移时，能按特性理解变更
- 避免 `自用特性/` 与生产目录双份实现漂移

## 四层真相源

| 层 | 位置 | 回答的问题 | 是否可运行代码 |
|---|---|---|---|
| 行为真相 | `自用特性/<中文特性名>/README.md` | 这个特性做什么、边界是什么、如何验收 | 否 |
| 定位真相 | `自用特性/<中文特性名>/CODEMAP.md` | 改了哪些文件/符号，怎么验证，哪里易冲突 | 否 |
| 代码真相 | 项目原有生产路径 | 当前真实实现是什么 | 是，唯一实现 |
| 变更真相 | Git 特性提交；可选 `patches/` | 相对上游改了什么、如何重放或回退 | patch 仅备份 |

原则：

1. **文档不落生产源码**
2. **生产实现只保留一份**
3. **Git 提交按特性拆分**
4. **`patches/` 只做灾备，不做日常开发主路径**

## 强制规则

### 文档

- 根目录 `AGENTS.md` 是执行约束。
- 所有新增或改变行为的自用特性，必须在 `自用特性/<中文特性名>/` 下维护。
- 每个目录至少包含：
  - `README.md`：目标、用户行为、关键决策、非目标、兼容/迁移、验收与回归
  - `CODEMAP.md`：真实文件、关键符号、数据流、易冲突点、验证命令
- 新建、合并、重命名特性时，同步更新 `自用特性/README.md`。
- 实现、测试、README、CODEMAP 必须同一任务内保持一致。
- 旧 `FEATURE/` 不再使用。

### 代码

- 生产源码留在项目原目录；`CODEMAP` 只记录真实路径。
- 禁止把可运行生产源码复制进 `自用特性/`。
- 允许放入截图、示例 JSON、迁移说明、一次性校验脚本等**说明材料**。
- 若需要展示关键差异，可在 README/CODEMAP 中放极小 diff 摘录，并标明来源提交；摘录不是第二实现。

### 提交

- 推荐提交前缀：
  - `feat(自用/<中文特性名>): ...`
  - `fix(自用/<中文特性名>): ...`
  - `docs(自用/<中文特性名>): ...`
- 一个提交尽量只服务一个自用特性，或明确相关的一组最小改动。
- 格式化、依赖升级、CI、发布流程与功能提交分离。
- 大特性可用 `self/<短名>` 或 `codex/<topic>` 分支开发，合入前整理为可读提交序列。

### 可选 patch 备份

- 目录建议：`patches/`
- 用途：灾备、迁 fork、人工审阅差异
- 导出示例：

```powershell
# 以已知上游基线为起点导出
git format-patch <upstream-baseline>..HEAD --output-directory patches
```

- 约束：
  - patch 不是日常编辑入口
  - 不得从 patch 目录反向维护业务逻辑
  - 升级完成后若继续保留 patch，应重导并淘汰过期文件

## 推荐日常工作流

### 新做/改一个自用特性

1. 先确认中文特性名；已有则复用目录。
2. 在生产路径实现代码。
3. 同步写/更新 `README.md` 与 `CODEMAP.md`。
4. 更新 `自用特性/README.md` 索引。
5. 按特性拆分提交：实现与文档可同提交，也可 `feat` + `docs` 相邻提交。
6. 跑该特性回归清单。

### 跟上游

1. 保存工作树。
2. 读总索引，确认仍需保留的特性。
3. 按各特性 CODEMAP 预判冲突热点。
4. `fetch` 上游后，优先把自用提交 `rebase` 到新上游。
5. 冲突按特性逐个解决，不“一把梭”混修。
6. 跑相关回归；更新被上游冲掉或改路径的文档。
7. 如使用 `patches/`，完成后重导。

### 回退某个特性

1. 用总索引与提交前缀定位该特性提交。
2. 评估是否可独立 revert。
3. 回退后删除或标记对应特性目录，并更新总索引。

## 目录约定

```text
AGENTS.md
自用特性/
  README.md                          # 总索引 + 上游对齐清单
  自用特性保存方案/
    README.md                        # 本方案
    CODEMAP.md
  <中文特性名>/
    README.md                        # 行为契约
    CODEMAP.md                       # 代码地图
    截图或示例...                    # 可选材料
patches/                             # 可选；灾备导出，不参与日常双写
  0001-xxx.patch
src/...                              # 唯一生产实现
```

## 非目标

- 不把 `自用特性/` 变成第二代码仓。
- 不要求每次格式化/注释/依赖维护都新建特性目录。
- 不强制永远保留 `patches/`；需要灾备时再导出。
- 不修改原作者通用用户手册或正式 CHANGELOG，除非自用发布说明明确需要。

## 验收与回归

- [x] 根目录 `AGENTS.md` 描述四层真相源、提交纪律、可选 patch 与上游对齐。
- [x] 本目录由「文档规范」扩展为「保存方案」。
- [x] 明确禁止复制生产源码进 `自用特性/`。
- [x] 总索引包含本方案，并写明升级/回退流程。
- [ ] 后续新特性提交持续使用 `feat|fix|docs(自用/<特性名>)` 前缀。
- [ ] 若启用 `patches/`，导出命令与基线记录在当时的发布说明或本方案补充段。
- [x] 「发布」定义写入 AGENTS + 本方案：远端 personal 包 + 本机 exe 替换。

## 自用「发布」定义

当用户说 **发布** 时，默认包含以下完整闭环（详见根目录 `AGENTS.md`）：

1. **推到 GitHub 自用分支**（按特性提交后 push）
2. **远端编译 / 发布**
   - tag：`v*-personal*`
   - Windows personal release
   - macOS **无签名** personal release
   - 取消误触发的全量签名 `Release`
3. **本机编译 exe 并替换已安装版本**
   - 构建：`pnpm tauri build`
   - 替换：`%LOCALAPPDATA%\\Programs\\CC Switch\\cc-switch.exe`
4. **回报** tag、Release 链接、本机替换结果

> 「发布」≠ 只 push / 只打 tag / 只跑 CI。调试运行（`pnpm dev`）不叫发布。
