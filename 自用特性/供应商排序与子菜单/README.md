# 供应商排序、子菜单与序号

> 文档状态：最后同步 2026-07-16

## 目标

让 Provider 列表的顺序可理解、可切换并且稳定：普通新增默认追加到最下面；主菜单下方增加列表子菜单；每张 Provider 卡片显示当前列表序号。Provider 上下文快捷操作也集中到该子菜单，避免挤占应用级主工具栏。

## 用户行为

子菜单提供「排序维度 + 升/降序」：

| 维度 | 规则 | 默认可拖拽 | 默认方向 |
| --- | --- | --- | --- |
| 自定义 | 使用持久化 `sortIndex`，新增项追加到末尾 | 是 | — |
| 创建时间 | 按 `createdAt`（创建/导入时间） | 否 | 降序（新→旧） |
| 最近使用 | 按用量统计 `lastUsedAt`（最近一次请求） | 否 | 降序（近→远） |
| 可用性 | 按模型拉取结果：可用 > 无模型 > 失败 > 跳过 > 未检测；成功项模型数多者优先 | 否 | 降序（可用优先） |
| 名称 | 按供应商名称 | 否 | 升序（A→Z） |

交互：

- 点击某一维度：切换到该维度；若再次点击同一维度，则翻转升/降序。
- - 按应用分别记忆 `sortKey` + `sortDirection`。

### 为什么不是「最新/最早」两个独立按钮

旧实现把创建时间拆成两个模式按钮，文案「最新/最早」容易被理解成：

- 最近用过 / 很久没用
- 或最近探测过

实际只是 `createdAt`。现改为一个「创建时间」维度 + 升/降序，语义更清晰。

### 最近使用 vs 可用性

- **最近使用**：看本地用量统计的最近请求时间，回答“我最近在用谁”。
- **可用性**：看 `/models` 探测结果，回答“谁现在还能拉到模型”。
- 二者数据源不同，**可用性排序不能替代最近使用**。

## 子菜单快捷操作

```text
[自定义 创建时间 最近使用 可用性 名称] [筛选框] [仅显示可用]  |  [S3上传][S3下载] | [批量探测][搜索][定位当前][筛选]  [数量]
```

- AppSwitcher、设置、添加等应用级入口继续留在主工具栏。
- S3 上传/下载、批量探测、搜索和定位属于当前 Provider 列表，显示在 sticky 子菜单右侧。
- 操作区通过 `toolbarActions` 插槽接入。
- 子菜单保持单行且允许横向滚动。
- 加载中或列表为空时仍显示子菜单和快捷操作。

## 序号

- 序号从 1 开始，跟随当前排序结果变化。
- 搜索过滤后保留原序号，不重编号。

## 新增落底与旧数据兼容

数据库插入缺少 `sortIndex` 的 Provider 时，在同一事务内执行：

1. 按旧版可见顺序为同应用内 `sort_index IS NULL` 的历史记录补齐序号。
2. 计算当前应用的最大 `sort_index`。
3. 新项落到末尾。

更新已有 Provider 时，输入缺少 `createdAt`/`sortIndex` 不会清空原值。

旧 localStorage `cc-switch-provider-sort-mode:*`：

- `newest` → 创建时间 + 降序
- `oldest` → 创建时间 + 升序
- `availability` / `name` / `manual` 映射到对应维度

## 验收与回归

- [ ] 五个维度按钮切换正确，并按应用记忆 key+direction。
- [ ] 「仅显示可用」勾选后只保留模型探测成功的供应商。
- [ ] 名称后筛选框复用搜索逻辑（名称/备注/网址），与仅显示可用可叠加。
- [ ] 同维度再次点击会翻转升/降序；再次点击同一维度可翻转升/降序。`r`n- [ ] 可用性排序顺序：可用 > 无模型 > 失败 > 跳过 > 未检测。
- [ ] 最近使用排序按 `lastUsedAt`，无记录者靠后（降序时）。
- [ ] 自定义模式可拖拽；其他模式禁用拖拽并提示。
- [ ] 旧库存在空 `sortIndex` 时，新 Provider 仍在所有旧项之后。
